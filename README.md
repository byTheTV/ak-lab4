## Язык программирования

Реализован подмножество **Lisp-подобного** синтаксиса: программа — одна или несколько **S-выражений** подряд; комментарии от `;` до конца строки игнорируются.

### Синтаксис (форма Бэкуса–Наура)

```ebnf
program   ::= form { form } ;

form      ::= atom | list ;

atom      ::= integer | string | symbol ;

list      ::= '(' { form } ')' ;

integer   ::= ['-'] digit { digit } ;

digit     ::= '0' | … | '9' ;

string    ::= '"' { char } '"' ;   (* с экранированием \" и \\ *)

symbol    ::= печатные символы кроме пробела, '(', ')', '"', ';' ;
```

Список `(f a b …)` трактуется как вызов или **спецформа**, если `f` — зарезервированное имя (`setq`, `defun`, `if`, `progn`, арифметика, `in`/`out`, обработчики прерываний и т.д.).

### Семантика

- **Стратегия вычислений:** аппликативный порядок; аргументы функций вычисляются до вызова; `if` вычисляет только выбранную ветку.
- **Области видимости:** глобальные имена, задаваемые `setq`, размещаются в фиксированных слотах данных; параметры `defun` — отдельные слоты; вложенных лексических окружений нет (имена глобальные + параметры текущей функции).
- **Типизация:** целые знаковые в диапазоне, допускаемом 24-битным немедленным операндом там, где он нужен; строковые литералы — только на этапе компиляции (сегмент `pstr` в данных).
- **Литералы:** целые и строки в коде; целые могут попадать в код как `PUSH_IMM`; строки кладутся в область **pstr** в DM, доступ через адреса, выдаваемые компилятором.

Описания достаточно, чтобы «выполнить» маленькую программу на бумаге, зная раскладку стека из раздела про систему команд.

## Организация памяти

Архитектура **Гарварда** (`harv` в варианте): раздельные **IM** (код) и **DM** (данные). Машинное слово **32 бита** по умолчанию для слова в IM и DM.

| Ресурс | Размер (слова) | Назначение |
|--------|----------------|------------|
| IM | 65536 | инструкции, векторы прерываний |
| DM | 65536 | стек, литералы, переменные `setq`, строки `pstr` |

**Стек** растёт вверх по адресам; дно стека `STACK_BASE = 0x1000` — первое свободное слово под вершину. Регистров «accumulator» нет: операнды на **стеке** (`stack` в варианте).

**Прерывания:** в IM по адресам `1 + k` лежит слово-вектор для линии `k` (ожидается `JMP` на обработчик). Адрес `0` — старт программы после загрузки.

**Отображение при компиляции:** код — последовательность слов в IM; статические ячейки переменных и pstr — в начале DM по схеме раскладки из транслятора; стек использует область от `STACK_BASE` и выше во время исполнения.

## Система команд

Модель — **стековая машина** с фиксированным набором команд; по духу ближе к **RISC**-подобному набору (фиксированная длина слова, простые операции), без переменной длины инструкции.

### Кодирование слова

Старший байт — **опкод**, младшие 24 бита — **операнд** (адрес, немедленник, маска порта и т.д.):

```text
 31 ─────────── 24 23 ─────────────── 0
┌──────────────────┬───────────────────┐
│      opcode      │      operand      │
└──────────────────┴───────────────────┘
```

Бинарное представление (`binary` в варианте): файлы `code.bin` / `data.bin` — **little-endian** 32-битные слова (см. загрузчик).

### Набор команд и такты

Полный цикл команды моделируется таблицей тактов в `Cpu` (см. `src/ak_lab4/cpu.py`, словарь `_TICKS`).

| Opcode (hex) | Смысл | Такты |
|--------------|--------|------|
| 00 | NOP | 1 |
| 01 | PUSH_IMM imm24 | 2 |
| 02 | DUP | 2 |
| 03 | DROP | 2 |
| 04 | LOAD | 4 |
| 05 | STORE | 4 |
| 06 | SWAP | 2 |
| 10…16 | ADD, SUB, MUL, DIV, MOD, EQ, SLT | 3…6 |
| 20 | JMP | 2 |
| 21 | JZ | 4 |
| 22 | CALL | 4 |
| 23 | RET | 3 |
| 30 | IN | 3 |
| 31 | OUT | 3 |
| 32 | HALT | 1 |
| 33 | EI | 1 |
| 34 | CLI | 1 |

### Ввод-вывод и прерывания (`port`, `trap`)

- Порты **IN/OUT** с номерами из `Port` (`DATA_IN`, `DATA_OUT`, `IRQ_STATUS`, `IRQ_EOI`) — см. `src/ak_lab4/isa/__init__.py`.
- **Trap:** внешнее расписание событий на тактах (`--schedule` JSON в симуляторе) поднимает линии IRQ; после инструкции доставляется один запрос, модель кладёт возвратный адрес и переходит по вектору.

### Усложнение: superscalar

- За один вызов `step` при `Cpu(..., superscalar=True)` возможна **параллельная выдача двух последовательных по PC** инструкций, если нет конфликта по данным на вершине стека и нет «опасных» опкодов.
- **Зависимости:** реализована консервативная функция `can_dual_issue(op0, op1)` (узкий набор пар: в основном `NOP`/`PUSH_IMM`/`DUP` в безопасных сочетаниях). Ветвления, `HALT`, `CALL`/`RET`, порты IRQ и т.п. переводят шаг в **одиночную** выдачу.
- **Такты при паре:** `ticks += max(такты первой, такты второй)` — отражает идею общей длительности «макро-такта».
- **Журнал:** при двойной выдаче строка содержит маркер `PAR` и два слова подряд, например:  
  `0	PAR	0	00000000	1	00000000	USR`
- По умолчанию `superscalar=False` — **golden-тесты** и старые сценарии не меняются. Включение: поле `Cpu` или флаг CLI `--superscalar` у `python -m ak_lab4.simulator`.
- **Производительность:** на последовательностях, где компилятор выдаёт подходящие пары (например, несколько `NOP`/`PUSH` подряд), число тактов меньше, чем в скалярном режиме; на реальных программах выигрыш зависит от кода (для сравнения можно прогнать один и тот же `code.bin`/`data.bin` с флагом и без и сравнить итоговый счётчик тактов в сообщении `HALT ticks=…`).

## Транслятор

Консольное приложение: модуль `ak_lab4.translator`.

**Вход:** путь к текстовому файлу с исходником (`.tv` и т.п.).

**Выход:** `code.bin` (по умолчанию), опционально `data.bin` (`--data-out`, если нужны строки/данные), опционально листинг `--listing`.

**Этапы:** лексический разбор S-выражений → AST → обход с генерацией слов IM/DM (раскладка `setq`, `defun`, строк, векторов прерываний).

Пример:

```bash
python -m ak_lab4.translator golden/hello/source.tv -o code.bin --data-out data.bin
```

## Модель процессора

Консольное приложение: `python -m ak_lab4.simulator`.

**Вход:** `code.bin`, `data.bin`; опции `--input`, `--log`, `--max-ticks`, `--schedule`, `--superscalar`.

**Выход:** байты в `DATA_OUT` (накапливаются в списке и печатаются как результат в тестах), при `--log` — **журнал** строк с тактом, PC, словом инструкции, режимом `USR`/`ISR` (и `PAR` при двойной выдаче).

Программная модель в `src/ak_lab4/cpu.py` согласована с таблицей тактов и семантикой стека; **схемы DataPath / ControlUnit** в смысле ТЗ — аппаратные чертежи (Quartus и т.д., пункт `hw` варианта) оформляются отдельно от этого репозитория и должны отражать регистры PC/SP, память, АЛУ, а не структуру файлов Python. В отчёт рекомендуется вставить изображения схем из каталога `hw` курса, когда они готовы.

## Тестирование

1. **Юнит-тесты** (`pytest`) — парсер, кодоген, отдельные части симулятора, в т.ч. [tests/simulator/test_superscalar.py](tests/simulator/test_superscalar.py) на двойную выдачу и блокировки по control.
2. **Golden-тесты** — каталоги `golden/<имя>/`: исходник `source.tv`, при необходимости `input.txt`, эталон вывода `expected_output.txt`. Прогон: трансляция → загрузка IM/DM → симуляция до `HALT` → сравнение байтов вывода с эталоном (см. [tests/test_golden_phase_a.py](tests/test_golden_phase_a.py)).

| Каталог | Назначение (по ТЗ) |
|---------|-------------------|
| `hello` | hello world |
| `cat` | копирование ввода в вывод |
| `hello_user_name` | диалог с именем |
| `sort` | сортировка (ввод чисел в формате варианта) |
| `pstr_two` | демонстрация строк **pstr** |
| `prob1` | алгоритм по варианту (Euler #4, тяжёлый по тактам) |

Журнал модели в golden не обязателен полностью (объём), проверяется **вывод программы**; при отладке журнал можно включать локально через `--log`.

### Пример цепочки

```bash
python -m ak_lab4.translator golden/hello_user_name/source.tv -o /tmp/code.bin --data-out /tmp/data.bin
python -m ak_lab4.simulator /tmp/code.bin /tmp/data.bin --input golden/hello_user_name/input.txt
```
